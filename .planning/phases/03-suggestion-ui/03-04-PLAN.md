From OverlayController.swift — the scroll monitor (lines 147-153):
```swift
scrollMonitor = NSEvent.addGlobalMonitorForEvents(matching: .scrollWheel) { [weak self] event in
    if let targetPID,
       let cgEvent = event.cgEvent,
       cgEvent.getIntegerValueField(.eventTargetUnixProcessID) == Int64(targetPID) {
        self?.dismiss()
    }
}
```
Problem: `event.cgEvent` is frequently nil for native Cocoa scroll events. The entire `if let cgEvent` chain short-circuits and `dismiss()` is never called.

From OverlayController.swift — repositionAfterAccept (lines 471-474):
```swift

    view.entries = newEntries
    view.needsDisplay = true
}
```
Problem: `newEntries` are computed with screen-coordinate rects translated by the OLD `windowOrigin` (captured at line 441). The window frame is never recalculated from the new entry union rect. Compare with `show()` lines 101-130 which correctly computes `windowRect` from union, translates entries, and calls `setFrame`.

From OverlayController.swift — show() pattern for window frame (lines 101-130):
```swift
let unionRect = entries.reduce(CGRect.null) { $0.union($1.hitRect) }
let padding: CGFloat = 4

// ... translate entries to window-local ...

view.frame = NSRect(origin: .zero, size: windowRect.size)

```
This is the correct pattern that `repositionAfterAccept` must replicate after rebuilding entries.

    Fix the scroll monitor in `show()` (around line 147) to dismiss on ANY scroll event while the overlay is visible, removing the unreliable `event.cgEvent` PID filter.

    **Root cause (from UAT diagnosis):** `NSEvent.addGlobalMonitorForEvents` returns NSEvent objects where `.cgEvent` is frequently nil for scroll wheel events in native Cocoa apps like Notes. The optional chain `event.cgEvent?.getIntegerValueField(...)` silently evaluates to nil, so `dismiss()` is never called.

    **Fix:** Replace the scroll monitor body with unconditional `self?.dismiss()`. Rationale: the overlay should dismiss on ANY scroll event while visible (not just scrolls in the target app). If the user scrolls anywhere, the underline positions are stale. This is simpler and more reliable than attempting to identify the scroll target. The existing TargetAppObserver already handles app-switch dismissal, so there is no gap in coverage.

    Replace lines 147-153:
    ```swift
    scrollMonitor = NSEvent.addGlobalMonitorForEvents(matching: .scrollWheel) { [weak self] event in
        self?.dismiss()
    }
    ```

    Remove the `let targetPID = self.targetAppPID` line (around line 146) that was only used by the old filter. The `targetAppPID` property itself is still used by `targetAppObserver.install(pid:)` so keep the property.

    No new unit test needed for this fix — the scroll monitor is an NSEvent global monitor that cannot be triggered in unit tests (no event loop). The fix is verified by UAT re-test. The existing `dismiss()` tests already cover that dismiss clears state correctly.

  <name>Task 2: Fix repositionAfterAccept — recalculate window frame from new entry rects</name>
  <files>OpenGram/SuggestionUI/OverlayController.swift, OpenGramTests/SuggestionUITests/OverlayControllerTests.swift</files>
  <behavior>
    - Test 1: After accepting a suggestion, remaining suggestions have updated scalar offsets (existing test `repositionShiftsOffsets`, confirm passes)
    - Test 2: After accepting a suggestion, overlay window frame is recalculated (new regression test — verify overlayWindow.frame changes after accept when entries move)
  </behavior>
  <action>
    Fix `repositionAfterAccept()` to recalculate the window frame and retranslate entries to window-local coordinates, matching the pattern used in `show()`.

    **Root cause (from UAT diagnosis):** `repositionAfterAccept()` builds `newEntries` using screen-coordinate rects and translates them by the OLD `overlayWindow.frame.origin`. But it never computes a new window frame from the union of new entry rects, never calls `overlayWindow.setFrame(...)`, and never updates `view.frame`. The entries are drawn at the wrong window-local positions because the window origin is stale.

    **Fix — replace lines 441 through 474** (from `var newEntries` to end of function) with:

    1. Build `newEntries` using RAW SCREEN coordinates (do NOT translate by old window origin yet):
       ```swift
       var screenEntries: [UnderlineEntry] = []
       // ... for each surviving suggestion, boundsValidator returns screen-coord rects ...
       // Build UnderlineEntry with screen-coord underlineRect and hitRect (no offset translation)
       ```

    2. After the loop, compute new window frame from union of screen-coord hit rects (same pattern as `show()`):
       ```swift
       let unionRect = screenEntries.reduce(CGRect.null) { $0.union($1.hitRect) }
       let padding: CGFloat = 4
       let newWindowRect = unionRect.insetBy(dx: -padding, dy: -padding)
       ```

    3. Translate entries to window-local coordinates:
       ```swift
       let localEntries = screenEntries.map { entry in
           UnderlineEntry(
               underlineRect: entry.underlineRect.offsetBy(dx: -newWindowRect.origin.x, dy: -newWindowRect.origin.y),
               hitRect: entry.hitRect.offsetBy(dx: -newWindowRect.origin.x, dy: -newWindowRect.origin.y),
               suggestion: entry.suggestion
           )
       }
       ```

    4. Update view and window:
       ```swift
       if let view = underlineView {
           view.entries = localEntries
           view.frame = NSRect(origin: .zero, size: newWindowRect.size)
           view.needsDisplay = true
       }
       overlayWindow.setFrame(newWindowRect, display: false)
       ```

    Remove the `let windowOrigin = overlayWindow.frame.origin` line (around line 441) that captured the old origin — it is no longer needed.

    **Regression test:** Add a test that verifies `overlayWindow.frame` is updated after `acceptSuggestion`. The test should:
    1. Create an OverlayController with a MockAXAccessor
    2. Set up two suggestions where accepting the first shifts the second
    3. Record `overlayWindow.frame` before accept (it will be .zero since show() isn't called in unit tests, but the reposition logic still runs)
    4. Call `acceptSuggestion` for the first suggestion
    5. Verify `suggestions.count == 1` (existing behavior) AND that the reposition logic executed without crash

    Note: A full window-frame-position assertion is impractical in unit tests because BoundsValidator queries real AX APIs. The meaningful verification is that the code path executes without error and the surviving suggestion is retained. The actual pixel-level repositioning is verified by UAT re-test.
  </action>
