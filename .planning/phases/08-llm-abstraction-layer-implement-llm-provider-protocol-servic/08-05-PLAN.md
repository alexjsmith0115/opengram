    - Test: Panel level is set to .floating so it appears above other windows
  </behavior>
  <action>
    Root cause: macOS apps with `.accessory` activation policy (LSUIElement=YES) have no foreground presence. `NSPanel.makeKeyAndOrderFront(nil)` silently does nothing because the app is not active and the window server ignores orderFront requests from background-only processes.

    Fix in `LLMSettingsPanel.show()`:
    1. Before `panel.makeKeyAndOrderFront(nil)`, add `NSApp.activate(ignoringOtherApps: true)` â€” this temporarily brings the app to the foreground so the window server honors the orderFront call.
    2. Set `panel.level = .floating` so the panel stays visible even if another app takes focus.
    3. Apply the same activation call in the early-return path (line 12) where an existing visible panel is re-focused.

    The fix is two lines added to the `show()` method. No API changes. No new types.

    Regression test: Create `OpenGramTests/LLMSettingsPanelTests.swift`. Since `LLMSettingsPanel` directly calls AppKit, the test should use a protocol-based approach or verify the panel's configuration properties. At minimum, test that after calling `show()`, the panel exists, is configured with `.floating` level, and `isReleasedWhenClosed` is false. Use Swift Testing (`@Test`, `#expect`).

<task type="auto" tdd="true">
  <name>Task 2: Fix LLM stream finishing task reference not stored</name>
  <files>OpenGram/CheckEngine/CheckOrchestrator.swift, OpenGramTests/CheckOrchestratorTests.swift</files>
  <behavior>
    - Test: When LLM provider returns results, onLLMFinished callback fires (stream completes, not indefinite block)
    - Test: Finishing task reference is retained until stream completes
    - Test: Stream completes even when results arrive slowly (existing SlowMockLLMProvider test should still pass)
  </behavior>
  <action>
    Root cause: Lines 74-79 in CheckOrchestrator.swift create a `Task.detached` that awaits all detached tasks then calls `continuation.finish()`. This task's reference is discarded immediately â€” it is not stored in any variable. While Swift's runtime typically keeps detached tasks alive, the lack of a stored reference is the diagnosed issue from UAT.

    Additionally, the `for await llmBatch in stream` loop on line 82 runs in the actor's own async context (the caller of `runCheck`). If that caller's task is cancelled, the for-await exits early without processing remaining results, and `onLLMFinished` may not fire.

    Fix in `CheckOrchestrator.runCheck()`:
    1. Store the finishing task in a local variable (`let finishingTask = Task.detached { ... }`) so the reference is retained for the scope of `runCheck`.
    2. After the `for await` loop, explicitly `_ = await finishingTask.value` to ensure the finishing task has fully completed before calling `onLLMFinished`. This is belt-and-suspenders â€” the for-await already waits for `continuation.finish()`, but the explicit await guarantees ordering.
    3. Wrap the `for await` loop in a `do { } catch { }` or use `withTaskCancellationHandler` to ensure `onLLMFinished` fires even if the consumer is cancelled. Simplest approach: after the for-await loop, unconditionally call `await onLLMFinished()` (which is already there on line 89 â€” just ensure the path is always reached).

    The fix is: (a) `let finishingTask = Task.detached { ... }` instead of bare `Task.detached { ... }`, (b) no other structural changes needed since the for-await already blocks until finish.

    Regression test: Add a test to CheckOrchestratorTests.swift that verifies `onLLMFinished` fires within a reasonable timeout (e.g., 5 seconds). Use the existing `SlowMockLLMProvider` pattern. The test should confirm that `onLLMFinished` is called (not that it times out). Use an `AsyncStream` or continuation pattern to await the callback. Name: `llmStreamCompletesAndFinishedCallbackFires`.
  </action>
