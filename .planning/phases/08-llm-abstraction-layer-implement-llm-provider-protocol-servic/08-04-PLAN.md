  <name>Task 1: Fix MenuBuilder NSObject inheritance for target-action dispatch</name>
  <files>OpenGram/Shell/MenuBuilder.swift, OpenGramTests/MenuBuilderTests.swift</files>
  <behavior>
    - Test: Creating a MenuBuilder, calling buildMenu(), setting onSettingsTapped, then performing the settings item selector — the callback fires.
    - Test: The settings menu item's target is the MenuBuilder instance and its action is settingsClicked.
  </behavior>
  <action>
**Root cause:** `MenuBuilder` is a plain Swift class (no `NSObject` inheritance) but uses `@objc` selector dispatch via `settingsItem.target = self` and `action: #selector(settingsClicked)`. Objective-C message dispatch requires an `NSObject` subclass. Without it, the target-action pair silently fails at runtime — the menu item appears but clicking it does nothing.

**Fix in `OpenGram/Shell/MenuBuilder.swift`:**

1. Change class declaration from `final class MenuBuilder` to `final class MenuBuilder: NSObject`. This is the only change needed — the `@objc private func settingsClicked()` and `settingsItem.target = self` already exist and are correct once the class inherits from NSObject.

**Regression test in `OpenGramTests/MenuBuilderTests.swift`:**

Add a test that verifies the settings callback actually fires when the selector is performed:

```swift
@Test("Settings item fires onSettingsTapped callback when selector is performed")
func settingsCallbackFires() {
    let builder = MenuBuilder()
    let menu = builder.buildMenu()
    var callbackFired = false
    builder.onSettingsTapped = { callbackFired = true }

    let settingsItem = menu.items[1]
    // Perform the target-action pair directly (simulates NSMenu dispatch)
    _ = settingsItem.target?.perform(settingsItem.action)

    #expect(callbackFired, "onSettingsTapped must fire when settings menu item action is performed")
}
```

Also add a test confirming the target is correctly set:

```swift
@Test("Settings item target is the MenuBuilder instance")

}
```

Do NOT change StatusBarController.swift — the callback assignment order (buildMenu then set onSettingsTapped) is correct because settingsClicked reads the property at click time, not at build time.

<task type="auto" tdd="true">
  <name>Task 2: Shield LLM network calls from parent task cancellation</name>
  <files>OpenGram/CheckEngine/CheckOrchestrator.swift, OpenGramTests/CheckOrchestratorTests.swift</files>
  <behavior>
    - Test: When the parent task running runCheck is cancelled after Harper completes but before LLM finishes, LLM results are still delivered (not dropped).
    - Test: When LLM is not configured, cancellation behavior is unchanged (onLLMFinished still called).
  </behavior>
  <action>
**Root cause:** In `CheckOrchestrator.runCheck()`, the `withTaskGroup` at line 58 creates structured child tasks for LLM checks. When the caller's `checkTask` is cancelled (e.g., user presses hotkey again triggering `forceCheckNow()` which calls `checkTask?.cancel()`), Swift structured concurrency propagates cancellation to all child tasks. This causes `URLSession.data(for:)` inside `LLMService.check()` to throw `CancellationError` immediately, killing in-flight HTTP requests regardless of the 15-second timeout.

**Fix in `OpenGram/CheckEngine/CheckOrchestrator.swift`:**

Wrap the LLM TaskGroup body in `withTaskCancellationHandler` is insufficient — structured concurrency still cancels. Instead, use `withCheckedContinuation` + `Task.detached` to break the structured concurrency parent-child relationship for the network call only. The simpler approach: wrap the entire LLM section using `withoutActuallyEscaping` is not available.

The cleanest fix: replace `withTaskGroup` with individual `Task.detached` calls that feed results back through an `AsyncStream`. This isolates LLM network calls from parent cancellation while preserving incremental delivery.

Specifically, replace lines 57-74 with:

```swift
// 3. Fire all enabled check types in parallel, shielded from parent cancellation (D-12, WR-04-gap)
let enabledChecks = config.enabledChecks
let (stream, continuation) = AsyncStream<[Suggestion]>.makeStream()

let detachedTasks = enabledChecks.map { checkType in
    Task.detached { [llm] in
        let result = await llm.check(
            text: text, type: checkType, harperSpans: harperSpans,
            config: config, apiKey: apiKey
        )
        continuation.yield(result)
    }
}

// Close stream once all detached tasks finish
Task.detached {
    for task in detachedTasks {
        _ = await task.value
    }
    continuation.finish()
}

// Deliver results incrementally as each check type completes (D-01)
for await llmBatch in stream {
    let deduped = Self.hardFilter(llmBatch, harperRanges: harperRanges)
    if !deduped.isEmpty {
        await onLLMBatch(deduped, context)
    }
}
```

Key points:
- `Task.detached` breaks the structured concurrency chain so parent cancellation does NOT propagate to URLSession calls.
- `AsyncStream` preserves the incremental delivery pattern (results arrive as each check type completes).
- The `for await` loop on the stream will still terminate when the stream finishes, keeping the existing flow where `onLLMFinished` is called after all LLM results are delivered.
- If the parent IS cancelled while consuming the stream, the `for await` loop exits early but the detached tasks complete independently (fire-and-forget for in-flight requests). This is the desired behavior: we don't crash or lose data, we just stop delivering to a UI that no longer cares.
